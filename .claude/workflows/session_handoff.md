# ワークフロー：セッション引き継ぎ

コンテキスト枯渇・意図的中断からの作業再開を確実にするためのプロトコル。

---

## 起動時プロトコル

新しいセッション開始時、または会話リセット後に必ず実行する。

```
1. SESSION.md を読む
2. 「進行中」タスクを確認する
3. 関連する Implementation_Plan.md のフェーズを確認する
4. 「次のステップ」の先頭から作業を再開する
```

> SESSION.md が存在しない場合は、ユーザーにタスクを確認してから新規作成する。

---

## 中断時プロトコル

### トークン逼迫を検知した場合

コンテキストウィンドウの圧迫を感知したら、作業継続より **SESSION.md の更新を優先** する。

```
1. 現在編集中のファイルを保存・コミット（未保存の変更を残さない）
2. SESSION.md を以下の内容で更新する：
   - 進行中タスクの現在ステータス（どこまで完了したか）
   - 中断箇所（例: Green フェーズ途中、〇〇関数の実装中）
   - 次のステップ（再開時に迷わないレベルで具体的に）
   - 直近の判断・メモ（このセッションで決定した事項）
```

### 意図的に中断する場合（フェーズ完了・作業切り替え等）

```
1. SESSION.md の「完了済み」を更新する
2. 次のタスクを「未着手」に追記する
3. 「最終更新」欄を記入する
```

---

## SESSION.md の記載規約

### 進行中タスクの粒度

再開時に「どのファイルの・どの関数を・どの状態まで」が分かるレベルで記載する。

```markdown
### 進行中
- [ ] `/auth/login` エンドポイント実装
  - TDD フェーズ: Green 実装中
  - 対象ファイル: `src/auth/login.ts`
  - 状態: `generateToken()` の実装が未完了
```

### 直近の判断・メモの記載基準

以下に該当するものは必ず記録する：

- 設計の選択肢を比較して決定したもの（例: JWT有効期限を24hに決定）
- ユーザーから口頭で指示された仕様変更
- 後続タスクに影響する技術的制約の発見

### 完了済みの管理

完了したタスクは削除せず `[x]` にして残す。
Implementation_Plan.md との整合確認に使う。

---

## 再開の確認コマンド

```bash
# 作業中ファイルの変更状況を確認
git status

# 直近のコミット履歴を確認
git log --oneline -10
```
