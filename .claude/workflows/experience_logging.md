# ワークフロー：経験値蓄積（Learning Log）

セッションで得た知見を次回以降に活かすためのプロトコル。
`/revise-claude-md` コマンドから呼び出される。

---

## 保存先の使い分け

| 知見の種類 | 保存先 | 理由 |
|---|---|---|
| プロジェクト固有（このコードベースに関するもの） | `.claude/learnings/YYYY-MM.md` | git管理でチームに共有できる |
| 汎用パターン（他プロジェクトでも使える） | Claude Codeグローバルメモリ | プロジェクト横断で活用できる |

---

## Step 1: プロジェクト固有知見の記録

`.claude/learnings/` ディレクトリに月別ファイルで記録する。

### ファイル命名規則
```
.claude/learnings/2026-03.md
```

### テンプレート

```markdown
# 学習ログ: YYYY-MM

## YYYY-MM-DD

### [タイトル（簡潔に）]

**状況**: （何をしようとしていたか）
**問題/発見**: （何が起きたか・気づいたか）
**解決/対応**: （どう対処したか）
**再現条件**: （この問題が起きる条件）
**次回への教訓**: （次回どう活かすか）
```

### 記録例

```markdown
## 2026-03-15

### portless で `api.myapp` が解決されない（WSL2）

**状況**: WSL2でportlessを使って api.myapp.localhost:1355 にアクセスしようとした
**問題/発見**: `.localhost` サブドメインがWSL2では自動解決されない
**解決/対応**: `sudo portless hosts sync` を実行して /etc/hosts に追記
**再現条件**: WSL2環境でportlessを初回セットアップ時
**次回への教訓**: WSL2では必ず初回に `sudo portless hosts sync` を実行する
```

## Step 2: 汎用パターンのグローバルメモリへの保存

他プロジェクトでも使える知見は、Claude Codeのメモリシステムに保存する。

### 保存すべき内容の判断基準

以下に当てはまる場合はグローバルメモリに保存する:
- 特定の技術スタック（Next.js, FastAPI等）全般に適用できる知見
- ツール（portless, git-cliff等）の使い方の発見
- TDDサイクルやエージェント活用のパターン改善

### 保存方法

Claude Codeのメモリファイル（`~/.claude/projects/[project-hash]/memory/`）に以下の形式で保存:

```markdown
---
name: [知見タイトル]
description: [一行説明]
type: feedback  # または project / reference
---

[知見の内容]

**Why:** [なぜ重要か]
**How to apply:** [どう活かすか]
```

---

## Step 3: CLAUDE.md の更新提案

以下に該当する場合、CLAUDE.md への反映をユーザーに提案する:

- **環境設定の追加**: 新しいセットアップ手順が判明した
- **禁止事項の追加**: やってはいけないことが判明した
- **ベストプラクティスの確立**: 試行錯誤の結果、最適な方法が確定した

提案は実行せず、変更差分をユーザーに見せて承認を得てから変更する。

## セッション終了時のチェックリスト

```
[ ] このセッションで「ハマったこと」はあったか？ → .claude/learnings/に記録
[ ] 設計判断をしたか？（AかBかで迷った末に決めた） → 理由込みでlearningsに記録
[ ] 他プロジェクトでも使えそうな知見はあるか？ → グローバルメモリに保存
[ ] CLAUDE.mdに追加・修正すべき内容はあるか？ → 提案する
[ ] SESSION.mdは最新状態か？ → 更新する
```
